## Overview

Create GitHub epics and standalone issues from items in `todo.md`, with full board reorganization: grouping standalone issues into epics, decomposing oversized items, analyzing dependencies, setting implementation order, and marking issues as `status:ready` when all readiness criteria are met.

## Purpose

The `board-fill` command transforms your todo list into a structured, dependency-aware project board:

- **Interactive triage**: Extracts scope, intent, unknowns, and acceptance criteria from you
- **Smart grouping**: Groups related items into epics, adopts orphans into thematic epics
- **Dependency analysis**: Builds dependency graphs, detects cycles, computes implementation order
- **Readiness assessment**: Ensures issues meet both structural and substantive criteria before marking ready
- **Board reorganization**: Every run reorganizes the entire board, not just new items

## When to Use

Run `board-fill` when:

- You have items in `todo.md` ready to convert to GitHub issues
- The board has standalone issues that should be grouped into epics
- Issues are missing dependency metadata or implementation order
- You want to assess which issues are truly ready for implementation
- The board needs reorganization and health diagnostics

## Workflow

<Steps>

### Read and Parse todo.md

1. **Parse todo items**
   - Reads `todo.md` at project root
   - Strips leading markers (`-`, `*`)
   - Presents numbered list

2. **Select items to process**
   - Choose all, specific numbers, or exclude specific numbers
   - Remaining items stay in `todo.md` for later

### Full Board Scan

1. **Fetch all open issues**
   - Excludes `meta:board-overview`
   - Retrieves labels, sub-issues, dependencies

2. **Categorize issues**
   - **Epic**: has sub-issues
   - **Sub-issue**: child of an epic
   - **Standalone**: neither parent nor child

3. **Present board health summary**
   - Missing metadata (labels, dependencies, acceptance criteria)
   - Potential epic grouping candidates
   - Total counts by category

### Duplicate Detection

1. **New items vs. existing board**
   - Semantic comparison using keywords
   - Classifies as: Duplicate / Partial overlap / No match

2. **Existing board cross-deduplication**
   - Compares existing issues pairwise
   - Prompts to merge, keep both, or convert to sub-issue

### Item-Level Triage Questioning

Before classification, extracts information needed for genuinely ready issues:

1. **Assess clarity across six dimensions**
   - Scope/Definition: What does "done" look like?
   - Value/Why: What user problem does this solve?
   - Unknowns/Spikes: Open design decisions or research needs?
   - External Blockers: Dependencies on external teams/services?
   - Size/Decomposition: Single deliverable or multiple?
   - User/Stakeholder: Who is affected? Current workaround?

2. **Triage by theme**
   - Batches items sharing area/domain
   - Asks only about unclear dimensions
   - Avoids repetitive questions

3. **Capture structured context**
   - Stores answers for use in classification, grouping, and acceptance criteria
   - Unresolved dimensions become readiness gaps later

### Issue Type & Executor Classification

Classifies each item using triage context and keyword signals:

1. **Type classification**
   - `type:bug`, `type:feature`, `type:refactor`, `type:qa`, `type:docs`, `type:infra`
   - Considers spike needs for research-heavy items

2. **Executor classification**
   - `executor:agent`: Clear criteria, bounded scope
   - `executor:human`: Decisions, infra, external setup
   - `executor:hybrid`: Agent implements after human direction

3. **Priority, area, and risk**
   - Priority: `p0` (critical/security) → `p3` (cosmetic)
   - Area: From `board.areas` in hatch.json or inferred from codebase
   - Risk: `low` (isolated) → `high` (architectural, security-critical)

### Retrieve Application Context

1. **Project documentation**
   - Reads specs, ADRs, design docs
   - Prefers TOC/headers first, expands relevant sections only

2. **Codebase exploration**
   - Understands current state of affected areas
   - Uses explore subagents or direct file reads

3. **Consult project learnings**
   - Matches by area and tags
   - Surfaces pitfalls, patterns, decisions

4. **External research** (when needed)
   - For items referencing external libraries/frameworks
   - Uses Context7 MCP or web research

### Propose Grouping (Epics vs. Standalone)

**Grouping philosophy**: Minimize standalone issues to near-zero. Group aggressively into epics.

1. **Group new items**
   - Absorb into existing epics first
   - Form new epics from 2+ items sharing connection
   - Adopt orphans into thematic epics
   - Standalone only as last resort

2. **Regroup existing standalone issues**
   - Applies same aggressive grouping philosophy

3. **Decomposition check**
   - Evaluates whether items are too large for single issue
   - Splits along area boundaries or acceptance criteria clusters
   - Uses triage context for decomposition guidance

### Dependency Analysis & Implementation Order

1. **Infer dependencies**
   - Explicit references: "Depends on", "Blocked by", "After #N"
   - Semantic analysis: producer/consumer relationships
   - Area overlap: soft dependencies for shared areas
   - Cross-epic dependencies

2. **Build dependency DAG**
   - Validates no cycles
   - Computes topological ordering

3. **Determine implementation order**
   - Within epics: topological order, then priority
   - Across board: interleave by priority at each depth
   - Identifies parallel lanes

4. **Present dependency graph and ordering**
   - Shows hard and soft dependencies
   - Highlights parallel work opportunities
   - Drafts `## Dependencies` and `## Implementation Order` sections

### Readiness Assessment

Every issue must pass both **structural** and **substantive** criteria before being marked `status:ready`.

**Structural criteria** (AI can resolve):
- `## Dependencies` section present
- Appears in implementation order
- All required labels present
- Acceptance criteria present
- Body follows template structure

**Substantive criteria** (requires user input):
- Acceptance criteria are specific and testable
- Scope is bounded (explicit out-of-scope section)
- No unresolved unknowns
- Effort is estimable

**Resolution process**:
1. Classifies each `status:triage` issue as Ready or Not Ready
2. Resolves structural gaps inline
3. Asks user for substantive gaps
4. Re-evaluates after user confirmation
5. Marks ready only when all criteria met

### Refine Issue Content

Issue bodies follow type-specific templates:

**Bug Report**: Summary, Steps to Reproduce, Expected/Actual Behavior, Acceptance Criteria, Scope, References, Dependencies

**Feature Request**: Summary, Motivation, Proposed Solution, Acceptance Criteria, Scope, References, Dependencies

**Refactor**: Summary, Current State, Proposed Change, Acceptance Criteria, Scope, References, Dependencies

**QA Validation**: Summary, Test Scope, Pass/Fail Criteria, Coverage Targets, Scope, Dependencies

**Epics**: Overview, Sub-issues checklist, Acceptance criteria, all labels

**Acceptance criteria sourcing** (priority order):
1. Triage Context (user's stated requirements)
2. User-stated edge cases or constraints
3. Codebase context (tests, interfaces, patterns)
4. AI inference (flagged for validation)

**Token-efficient presentation**:
- Compact summary table for all issues
- Full details only on request
- Flags issues needing attention

### Create & Update Issues

1. **Create new issues**
   - Executes in dependency order (parents before children)
   - Links sub-issues to epics
   - Adds to project board with status sync

2. **Update existing issues**
   - Adds/updates `## Dependencies`
   - Regenerates `## Implementation Order` for epics
   - Applies epic regrouping
   - Marks `status:ready` (never downgrades)
   - Syncs Projects v2 status

3. **Present summary**
   - New issues created
   - Existing issues updated
   - Board summary statistics

### Refresh Board Dashboard

**Mandatory step** - regenerates the `meta:board-overview` issue:

1. Computes implementation lanes
2. Assigns models to all open issues
3. Updates dashboard with current board state
4. Creates dashboard if it doesn't exist

### Cleanup

Prompts to remove processed items from `todo.md`:
- Remove all, only created ones, or keep all
- Preserves skipped/excluded lines

</Steps>

## Configuration in hatch.json

The command reads configuration from `/.agents/hatch.json`:

```json
{
  "owner": "your-org",
  "repo": "your-repo",
  "board": {
    "owner": "your-org",
    "repo": "your-repo",
    "defaultBranch": "main",
    "projectNumber": 1,
    "areas": ["frontend", "backend", "infra"],
    "labels": {
      "type": ["type:bug", "type:feature", "type:refactor"],
      "executor": ["executor:agent", "executor:human", "executor:hybrid"],
      "status": ["status:triage", "status:ready", "status:in-progress"],
      "priority": ["priority:p0", "priority:p1", "priority:p2", "priority:p3"],
      "risk": ["risk:low", "risk:med", "risk:high"]
    }
  }
}
```

## Integration with GitHub Agentic Workflows

hatch3r's board commands operate as the **implementation orchestration layer** above GitHub Agentic Workflows:

- **board-init** sets up the project management structure
- **board-fill** creates the work items that agentic workflows can triage and label
- **board-pickup** orchestrates the implementation → review → merge pipeline

Use agentic workflows for continuous background automation, use hatch3r board commands for structured delivery orchestration.

## Error Handling

- **search_issues failure**: Retries once, warns and proceeds without deduplication
- **issue_write failure**: Reports error, skips that issue, continues with others
- **sub_issue_write failure**: Reports error but does not delete created issue
- **Dependency cycles**: Flags and requires resolution before proceeding

## Guardrails

- Never creates issues for duplicate topics without explicit approval
- Never skips ASK checkpoints
- Uses correct labels from taxonomy defined in `hatch.json`
- Keeps issue bodies concise and grounded in user requirements
- Detects and prevents dependency cycles
- Never downgrades issue status (only upgrades `triage` → `ready`)
- Always performs full board scan
- Preserves existing issue content when appending sections
- Board Overview is auto-maintained and excluded from analysis
