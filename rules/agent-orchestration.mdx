# Agent Orchestration

**Scope:** Always applied  
**Rule ID:** `hatch3r-agent-orchestration`

Defines mandatory agent delegation, skill loading, and subagent usage directives for ALL tasks in ALL contexts. These are not suggestions — they are requirements.

## Universal Applicability

This rule applies to **every context** without exception:

- Board-pickup (epic, sub-issue, standalone, batch)
- Workflow command (full mode and quick mode)
- Plain chat (single task or multiple tasks)
- Issue references (e.g., "implement #5")
- Natural language requests (e.g., "add a dark mode toggle")

Whether the user invokes a command or simply asks for a task in conversation, the full sub-agent pipeline is mandatory. There is no context where implementing code inline (without sub-agents) is acceptable.

## Universal Sub-Agent Pipeline

Every task MUST follow this three-phase pipeline:

### Phase 1 — Research

Spawn `hatch3r-researcher` for context gathering.

**Skip only for:**
- Trivial single-line edits (typos, comment fixes, single-value config changes)

**All other tasks require researcher context.**

**Research mode selection:**

| Task Type | Research Modes |
|-----------|----------------|
| `type:bug` | `symptom-trace`, `root-cause`, `codebase-impact` |
| `type:feature` | `codebase-impact`, `feature-design`, `architecture` |
| `type:refactor` | `current-state`, `refactoring-strategy`, `migration-path` |
| `type:qa` | `codebase-impact` |

**Depth selection:**
- `quick` — Low-risk tasks
- `standard` — Medium-risk tasks  
- `deep` — High-risk tasks

### Phase 2 — Implement

Spawn `hatch3r-implementer` for ALL code changes.

**One dedicated implementer per task.**

**Never implement inline — always delegate via the Task tool.**

**Task scenarios:**

| Scenario | Implementation Strategy |
|----------|------------------------|
| Single standalone issue | Spawn one `hatch3r-implementer`. Orchestrator coordinates git, PR, and board operations. |
| Plain chat single task | Spawn one `hatch3r-implementer`. Create synthetic issue context first (title, acceptance criteria, type). |
| Epics with sub-issues | Spawn one `hatch3r-implementer` per sub-issue. Execute level-by-level respecting dependency order. |
| Multiple standalone issues (batch) | Group by dependency level, spawn one `hatch3r-implementer` per issue, execute level-by-level. Shared branch, combined PR. |

### Phase 3 — Quality

Spawn ALL applicable specialists in parallel after implementation:

| Specialist | When | Mandatory? |
|-----------|------|------------|
| `hatch3r-reviewer` | After every implementation | **YES** — always |
| `hatch3r-test-writer` | After every code change | **YES** — always for code changes |
| `hatch3r-security-auditor` | After every code change | **YES** — always for code changes |
| `hatch3r-docs-writer` | After every implementation | **EVALUATE** — spawn when changes affect APIs, architecture, user-facing behavior, or when specs/ADRs need updating |
| `hatch3r-lint-fixer` | When lint errors present | Conditional |
| `hatch3r-a11y-auditor` | When UI/accessibility changes | Conditional |
| `hatch3r-perf-profiler` | When performance-sensitive changes | Conditional |
| `hatch3r-dependency-auditor` | When dependencies change | Conditional |
| `hatch3r-ci-watcher` | When CI fails | Conditional |

**Launch as many independent subagents in parallel as the platform supports — no artificial concurrency limit.**

## Agent Roster

### Core Agents (Always Used)

**`hatch3r-researcher`**

- **Purpose:** Context gathering across 12 research modes
- **Invoke when:** ALWAYS before implementation (skip only for trivial single-line edits)
- **Research modes:** `symptom-trace`, `root-cause`, `codebase-impact`, `feature-design`, `architecture`, `current-state`, `refactoring-strategy`, `migration-path`, and more

**`hatch3r-implementer`**

- **Purpose:** Focused single-task implementation
- **Invoke when:** ALWAYS for code changes
- **Protocol:** One dedicated implementer per task (standalone issues, epic sub-issues, batched issues, plain chat tasks)

### Quality Specialists (Phase 3)

**`hatch3r-reviewer`**

- **Purpose:** Code review for quality, security, performance
- **Invoke when:** ALWAYS after implementation, before PR creation
- **Input:** Diff and acceptance criteria

**`hatch3r-test-writer`**

- **Purpose:** Regression and coverage tests
- **Invoke when:** ALWAYS for code changes (not just bugs — every code change gets tests)
- **Output:** Unit tests, regression tests, integration tests as applicable

**`hatch3r-security-auditor`**

- **Purpose:** Security rules, data flows, access control
- **Invoke when:** ALWAYS for code changes (not just `area:security` — every code change gets a security review)
- **Output:** Security audit report with findings and recommendations

**`hatch3r-docs-writer`**

- **Purpose:** Specs, ADRs, documentation maintenance
- **Invoke when:** ALWAYS evaluate. Spawn when changes affect APIs, architecture, or user-facing behavior.
- **Output:** Updated specs, ADRs, or documentation files

### Conditional Specialists

**`hatch3r-lint-fixer`**

- **Invoke when:** Lint or type errors present after implementation
- **Purpose:** Style, formatting, type error cleanup

**`hatch3r-a11y-auditor`**

- **Invoke when:** UI or accessibility changes made
- **Purpose:** WCAG AA compliance checks

**`hatch3r-perf-profiler`**

- **Invoke when:** Performance-sensitive changes made
- **Purpose:** Performance profiling and optimization

**`hatch3r-dependency-auditor`**

- **Invoke when:** Dependencies change or new packages added
- **Purpose:** Supply chain security, CVE scanning

**`hatch3r-ci-watcher`**

- **Invoke when:** CI fails during or after implementation
- **Purpose:** CI/CD failure diagnosis and fix suggestions

## Skill Loading Directives

Before implementing any task, you MUST read and follow the matching hatch3r skill:

| Task Type | Skill |
|-----------|-------|
| `type:bug` | `hatch3r-bug-fix` |
| `type:feature` | `hatch3r-feature` (aka `hatch3r-feature-implementation`) |
| `type:refactor` + `area:ui` | `hatch3r-visual-refactor` |
| `type:refactor` + behavior change | `hatch3r-logical-refactor` |
| `type:refactor` (other) | `hatch3r-refactor` (aka `hatch3r-code-refactor`) |
| `type:qa` | `hatch3r-qa-validation` |

When a skill references agents under "Required Agent Delegation", those delegations are mandatory — you MUST spawn the listed agents via the Task tool.

## Subagent Spawning Protocol

When spawning any subagent via the Task tool:

### 1. Use Correct Subagent Type

```typescript
subagent_type: "generalPurpose"
```

Use for all hatch3r agent delegations.

### 2. Include in Every Subagent Prompt

- The agent protocol to follow (e.g., "Follow the hatch3r-implementer agent protocol")
- All `scope: always` rules from `/.agents/rules/` that apply
- The project's tooling hierarchy (Context7 MCP for library docs, web research for current context)
- Relevant learnings from `/.agents/learnings/` if the directory exists

### 3. Maximum Parallelism

**Launch as many independent subagents in parallel as the platform supports.**

Do not impose an artificial concurrency limit. Use maximum parallelism for independent work.

### 4. Await and Review Results

Before proceeding:

- Await all subagent completions
- Review results for quality and completeness
- If a subagent reports BLOCKED or PARTIAL, surface to the user

## Single-Task Plain Chat Protocol

When the user provides a single task in plain chat (no command invoked, no issue reference), the full sub-agent pipeline still applies:

### Process

1. **Classify** the task by type (bug/feature/refactor/QA/other) based on context
2. **Create synthetic issue context** — title, acceptance criteria, and type — from the user's instruction
3. **Run the Universal Sub-Agent Pipeline:** Phase 1 (Research) → Phase 2 (Implement) → Phase 3 (Quality)
4. For GitHub issue references in chat (e.g., "fix #5"), fetch issue details via `gh issue view` and use them as the task context instead of creating synthetic context

This ensures consistent quality regardless of how the task was initiated.

## Multi-Task Detection (Plain Chat)

When the user provides multiple tasks in a single message:

- Numbered lists
- Comma-separated instructions
- Multiple issue references (e.g., "implement #1, #3, #7")
- Multiple distinct requests

**You MUST parallelize them:**

### Process

1. **Parse** the message into individual discrete tasks (each distinct implementation request is one task)
2. **Classify** each task by type (bug/feature/refactor/QA/other) based on context or explicit labels
3. **Build a dependency graph** among the tasks. Independent tasks share the same level and run in parallel.
4. **Spawn one `hatch3r-researcher` subagent per task** (skip for trivial single-line edits only). Launch in parallel.
5. **Spawn one `hatch3r-implementer` subagent per task** per dependency level
6. **For GitHub issue references:** fetch issue details via `gh issue view` and use them as the task context
7. **For natural language tasks:** create synthetic issue context (title, acceptance criteria, type) from the instruction. Pass this context to the implementer subagent.
8. **Spawn the full quality pipeline** after all implementations complete: reviewer + test-writer + security-auditor (always), plus docs-writer, auditors as applicable

This directive applies regardless of whether board-pickup was invoked. Any context where implementation tasks are identified MUST use one subagent per task with maximum parallelism.

## Example: Single Feature Request

**User:** "Add a dark mode toggle to the settings page"

**Orchestrator process:**

1. Classify: `type:feature`
2. Create synthetic context:
   - Title: "Add dark mode toggle to settings page"
   - Acceptance criteria: "User can toggle dark mode from settings page. Preference persists across sessions."
   - Type: `feature`
3. Load skill: `hatch3r-feature`
4. Phase 1: Spawn `hatch3r-researcher` with modes: `codebase-impact`, `feature-design`, `architecture`
5. Phase 2: Spawn `hatch3r-implementer` with research context and acceptance criteria
6. Phase 3 (parallel): Spawn `hatch3r-reviewer`, `hatch3r-test-writer`, `hatch3r-security-auditor`, `hatch3r-a11y-auditor` (UI change), and evaluate `hatch3r-docs-writer`
7. Review results, create PR, update board

## Example: Multiple Issues

**User:** "Implement #12, #15, and #23"

**Orchestrator process:**

1. Fetch issue details: `gh issue view 12`, `gh issue view 15`, `gh issue view 23`
2. Classify each by labels/content: #12 (`bug`), #15 (`feature`), #23 (`feature`)
3. Build dependency graph (assume all independent)
4. Phase 1 (parallel): Spawn `hatch3r-researcher` for #12, #15, #23
5. Phase 2 (parallel): Spawn `hatch3r-implementer` for #12, #15, #23
6. Phase 3 (parallel): Spawn reviewer, test-writer, security-auditor for each (9 subagents total), plus conditional specialists
7. Review results, create combined PR, update board for all three issues

## Rule Application

All hatch3r rules with `scope: always` apply to every implementation task, including work delegated to subagents.

**When constructing subagent prompts:**

- Include the rule directives
- Subagents do not automatically inherit the parent's rule context
- Explicitly pass relevant rules in the subagent prompt

## Enforcement

**Code review checklist:**
- [ ] Research phase completed (or justified skip for trivial edit)
- [ ] Implementation delegated to subagent (no inline implementation)
- [ ] All mandatory quality specialists invoked (reviewer, test-writer, security-auditor)
- [ ] Conditional specialists invoked when triggered
- [ ] Results reviewed before PR creation

**Workflow violations:**

- Inline implementation without subagent — Reject, require re-implementation via subagent
- Skipped quality specialists — Reject, require full quality pipeline
- Missing research phase — Reject (unless trivial edit)
