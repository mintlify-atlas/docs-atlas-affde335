# Rules Overview

Hatch3r enforces 22 engineering rules that automatically guide code quality, architecture, security, and operations. These rules apply to all agents, commands, and workflows without exception.

## How Rules Work

**Scope-based application:**
- `scope: always` — Applied to every task in every context (chat, board pickup, workflows, commits)
- `scope: conditional` — Applied when file globs match or context triggers the rule

**Rule integration points:**
- Loaded before every implementation task begins
- Enforced during code review by `hatch3r-reviewer`
- Validated during security audits by `hatch3r-security-auditor`
- Checked during test generation by `hatch3r-test-writer`
- Referenced in skill protocols and agent orchestration

**Rule priority:**
1. Security rules override all other considerations
2. Architectural rules prevent tech debt accumulation
3. Code quality rules ensure maintainability
4. Operational rules enable reliable deployments

### Additional Standards (10 rules)

These rules apply conditionally based on context:

- **Accessibility Standards** — WCAG 2.2 AA compliance, keyboard navigation, screen reader support
- **Browser Verification** — Visual verification protocol for UI changes using browser automation
- **CI/CD Standards** — Pipeline design, stage gates, deployment strategies, and rollback procedures
- **Data Classification** — PII handling, encryption requirements, retention policies, and GDPR/CCPA
- **Internationalization** — ICU MessageFormat 2.0, RTL support, locale management, and layout accommodations
- **Learning Consultation** — Auto-consult project learnings before implementation to avoid past pitfalls
- **Performance Budgets** — Core Web Vitals, API response times, database query limits, and enforcement
- **Secrets Management** — Env var management, rotation policies, cloud secret managers, and emergency procedures
- **Theming** — Dark mode, high contrast support, semantic color tokens, and theme detection
- **Tooling Hierarchy** — GitHub CLI over MCP, documentation MCP for libraries, web research for current context

## Rule Enforcement

**Automated gates:**
- Lint and type errors block commits (pre-commit hooks)
- Test coverage regressions block PR merge (CI stage gate)
- Security vulnerabilities block deployment (dependency scan gate)
- Bundle size violations block merge (size-limit in CI)
- Core Web Vitals regressions trigger alerts (Lighthouse CI)

**Agent-enforced quality:**
- `hatch3r-reviewer` validates adherence to code standards and architectural rules
- `hatch3r-security-auditor` checks security patterns and data classification
- `hatch3r-test-writer` ensures coverage thresholds and test quality standards
- `hatch3r-a11y-auditor` validates accessibility standards for UI changes

**Human oversight:**
- PR review required for rule exceptions (documented with issue links)
- Tech lead approval for performance budget overrides
- Security team review for data classification changes

## Customizing Rules

Use `/hatch3r rule-customize` to create project-specific rule variants:

```bash
/hatch3r rule-customize code-standards --add "Max cyclomatic complexity: 15" --reason "Legacy codebase migration"
```

Customizations are tracked in `/.agents/rules/` and override base rules while maintaining the core structure.
