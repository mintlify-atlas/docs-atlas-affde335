# Supported Tools

hatch3r maintains a **single canonical source** in `/.agents/` and generates **native configuration** for 13 AI coding platforms. You define your agent setup once and hatch3r adapts it to whatever tool you use.

## Platform Overview

All 13 platforms are supported with varying levels of capability:

| Tool | Rules | Agents | Skills | Commands | MCP | Hooks |
|------|:-----:|:------:|:------:|:--------:|:---:|:-----:|
| **Cursor** | Y | Y | Y | Y | Y | Y |
| **GitHub Copilot** | Y | Y | Y | Y | Y | -- |
| **Claude Code** | Y | Y | Y | Y | Y | Y |
| **Cline / Roo Code** | Y | Y | Y | Y | Y | Y |
| **OpenCode** | Y | Y | Y | Y | Y | -- |
| **Codex CLI** | B | B | Y | -- | Y | -- |
| **Gemini CLI** | Y | B | Y | Y | Y | Y |
| **Windsurf** | Y | B | Y | Y | Y | -- |
| **Amp** | B | B | Y | ~ | Y | -- |
| **Aider** | B | B | Y | -- | -- | -- |
| **Kiro** | Y | B | Y | -- | Y | -- |
| **Goose** | B | B | B | -- | -- | -- |
| **Zed** | B | B | -- | -- | -- | -- |

**Legend:**
- **Y** = Adapter emits files for this capability
- **B** = Bridge (content folded into an instruction file)
- **~** = Platform reads canonical `.agents/` paths natively
- **--** = No platform support

<Note>
  hatch3r automatically selects the right adapter based on your tool selection during `npx hatch3r init`.
</Note>

## Adapter Details by Capability

### Rules

All adapters emit rules in some form:

- **Separate files:** Cursor, Copilot, Claude, Cline, Windsurf, OpenCode, Kiro
- **Bridge (inlined):** Codex, Gemini, Amp, Aider, Goose, Zed

**18 included rules:**
- `code-standards` - TypeScript strict mode, naming conventions, file length limits
- `error-handling` - Structured errors, typed error codes, retries, correlation IDs
- `testing` - Deterministic, isolated tests with clear naming
- `api-design` - Endpoint versioning, validation, idempotency, auth
- `component-conventions` - Component structure, typed props, WCAG AA accessibility
- `security-patterns` - Input validation, output encoding, auth enforcement
- `observability` - Structured logging, OpenTelemetry, SLO/SLI
- `performance-budgets` - Core Web Vitals, API latency, bundle size
- `theming` - Dark mode, CSS custom properties, semantic tokens
- `i18n` - Internationalization, RTL support, ICU message format
- `feature-flags` - Flag naming, evaluation, gradual rollout
- `migrations` - Backward-compatible schema changes, rollback plans
- `dependency-management` - Lockfile hygiene, CVE patching, bundle budgets
- `git-conventions` - Branch naming, commit messages, merge strategy
- `browser-verification` - When/how to verify UI via automation MCP
- `agent-orchestration` - Sub-agent spawning, result aggregation, coordination
- `learning-consult` - When to consult project learnings
- `tooling-hierarchy` - Priority order: specs > codebase > library docs > web

## Model Selection

All adapters support per-agent model customization:

- **Native support:** Cursor, Copilot (VS Code), OpenCode, Codex
- **Guidance (text-based):** Claude, Cline, Gemini, Windsurf, Amp, Aider, Kiro, Goose, Zed

Configure via:
1. `hatch.json` - Global default and per-agent overrides
2. Agent frontmatter - `model:` field in canonical agent files
3. `.hatch3r/agents/{id}.customize.yaml` - Per-agent customization

**Resolution order:** customization file > manifest per-agent > agent frontmatter > manifest default

See the [Model Selection guide](/guides/model-selection) for details.
