## Overview

The performance audit skill guides agents through profiling, identifying violations, and implementing optimizations against defined performance budgets.

## Quick Start

```
Task Progress:
- [ ] Step 1: Read performance budgets from rules and specs
- [ ] Step 2: Profile — bundle size, runtime, memory
- [ ] Step 3: Identify violations — which budgets exceeded, which hot paths slow
- [ ] Step 4: Plan optimizations — code splitting, lazy loading, memoization, etc.
- [ ] Step 5: Implement optimizations with before/after measurements
- [ ] Step 6: Verify all budgets met, no regressions
```

## Step 1: Read Performance Budgets

Load the project's performance budgets from project rules and quality documentation:

### Common Metrics

- **Render frame rate** - Target 60fps (16ms/frame)
- **Cold start to interactive** - Time to interactive (TTI)
- **Idle CPU usage** - CPU load when idle
- **Memory footprint** - Heap size limits
- **Event processing latency** - Response time for interactions
- **Bundle size** - Initial and gzipped sizes
- **Database reads per session** - Query optimization targets
- **API warm execution** - Backend response times

Note which surface is in scope: frontend, backend, or both.

## Step 2: Profile

### Bundle Size

- Run `npm run build` and inspect output for gzipped sizes
- Use `vite-bundle-visualizer`, `rollup-plugin-visualizer`, or `webpack-bundle-analyzer` to identify large chunks and dependencies
- Compare bundle sizes if multiple build targets exist

### Runtime (Frontend)

- Use Chrome DevTools Performance tab: record startup, record key interactions
- Measure: Time to Interactive (TTI), First Contentful Paint (FCP), Largest Contentful Paint (LCP)
- Use **cursor-ide-browser MCP** `browser_profile_start`/`browser_profile_stop` for CPU profiling with call stacks
- Check frame rate during animations (target: 60fps, 16ms/frame)

### Memory

- Heap snapshot before/after session (target per project budget)
- Look for leaks: detached DOM, growing arrays, uncleared timers

### Backend/API

- Check monitoring for cold start and warm execution times
- Instrument key paths

## Step 3: Identify Violations

- List which budgets are exceeded and by how much
- Identify hot paths: which functions/components contribute most to latency or bundle size
- Prioritize: critical user flows first
- Document baseline measurements for comparison

## Step 4: Plan Optimizations

### Common Strategies

#### Code Splitting

- Route-based or component-based splitting
- Lazy-load panels, modals, non-critical features

#### Lazy Loading

- `defineAsyncComponent`, dynamic `import()` for heavy components

#### Memoization

- `computed`, `memo` for expensive derivations
- Avoid unnecessary re-renders

#### Reduce Re-renders

- `v-show` over `v-if` for frequently toggled elements
- `shallowRef` where appropriate

#### Bundle Optimization

- Remove unused deps
- Replace heavy libs with lighter alternatives
- Tree-shake unused code

#### Images/Assets

- Optimize, lazy-load, use appropriate formats

#### Database

- Reduce reads (batch, cache, denormalize)

#### Cloud/API

- Warm-up strategies
- Reduce cold starts

### Constraints

- Check project ADRs for architectural constraints
- Ensure optimizations don't violate privacy/security invariants
- Respect `prefers-reduced-motion` - do not add animations that ignore it

## Step 5: Implement Optimizations

- Apply changes incrementally
- Measure before and after each change
- Document before/after for each metric in PR or audit report
- Run full test suite after each optimization to avoid functional regressions

### Measurement Protocol

1. Take baseline measurement
2. Apply single optimization
3. Take new measurement
4. Document delta
5. Commit if improvement, revert if regression
6. Repeat for next optimization

## Step 6: Verify

```bash
npm run lint && npm run typecheck && npm run test
npm run build
```

- [ ] All performance budgets met
- [ ] No functional regressions
- [ ] Before/after measurements documented
- [ ] CI performance check passes (if configured)
