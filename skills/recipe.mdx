## Overview

The recipe management skill guides you through creating reusable workflow recipes that compose hatch3r commands, skills, and agents into repeatable sequences.

## Quick Start

```
Task Progress:
- [ ] Step 1: Identify the workflow to capture as a recipe
- [ ] Step 2: Design the step sequence and dependency graph
- [ ] Step 3: Write the recipe YAML
- [ ] Step 4: Test with dry-run mode
- [ ] Step 5: Validate with a real execution
```

## Step 1: Identify Workflow

Determine the repeatable workflow pattern:

- What commands/skills/agents are involved?
- What order do they execute in?
- Which steps can run in parallel?
- Where should the user be asked to confirm (checkpoints)?

## Step 2: Design Step Sequence

Map out the dependency graph:

- List all steps with their hatch3r command or skill reference
- Identify dependencies between steps
- Identify steps that can run in parallel
- Mark checkpoint steps where user confirmation adds value

## Step 3: Write Recipe YAML

Create the recipe file in `.hatch3r/recipes/` following the schema defined in the `hatch3r-recipe` command. Include:

- Clear name and description
- Required variables with descriptions
- Steps with proper `depends_on` and `parallel_with` relationships
- Checkpoint markers at decision points
- Completion message with next steps

### Example Recipe Structure

```yaml
name: feature-to-production
description: Complete workflow from feature branch to production deploy

variables:
  - name: feature_branch
    description: Name of the feature branch
    required: true
  - name: issue_number
    description: GitHub issue number
    required: true

steps:
  - id: implement
    command: hatch3r-feature
    description: Implement the feature
    variables:
      issue: "{{issue_number}}"

  - id: qa
    skill: hatch3r-qa-validation
    description: Run E2E validation
    depends_on: [implement]
    checkpoint: true

  - id: pr
    command: hatch3r-pr-creation
    description: Create pull request
    depends_on: [qa]

  - id: release
    command: hatch3r-release
    description: Cut release and deploy
    depends_on: [pr]
    checkpoint: true

completion:
  message: "Feature deployed to production. Monitor for 24h."
```

## Step 4: Test with Dry-Run

Execute `--dry-run` to validate:

- YAML schema is valid
- All referenced commands/skills exist
- Dependency graph has no cycles
- Variables are properly referenced
- Prerequisites are checkable

```bash
npx hatch3r recipe run feature-to-production --dry-run
```

## Step 5: Validate with Real Execution

Run the recipe on a test project to verify:

- Steps execute in correct order
- Parallel steps don't conflict
- Checkpoints pause appropriately
- Error handling works (intentionally fail a step)
- Completion message is accurate

```bash
npx hatch3r recipe run feature-to-production \
  --var feature_branch=feat/test \
  --var issue_number=42
```

## Recipe Best Practices

### Use Checkpoints Wisely

- Before destructive operations (deploy, release)
- After validation steps (QA, security scan)
- When manual review adds value

### Design for Parallelism

- Identify independent steps that can run concurrently
- Use `parallel_with` for performance gains
- Ensure parallel steps don't conflict on file access

### Handle Failures Gracefully

- Define rollback steps for critical operations
- Use `on_failure` hooks for cleanup
- Provide clear error messages

### Document Variables

- Clear descriptions for all variables
- Examples in the description
- Mark required vs optional
