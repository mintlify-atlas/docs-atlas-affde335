## Overview

The dependency audit skill guides agents through security vulnerability remediation, package updates, and bundle size optimization.

## Quick Start

```
Task Progress:
- [ ] Step 1: Run npm audit + npm outdated, categorize findings
- [ ] Step 2: Research CVEs via web search for critical/high
- [ ] Step 3: Plan upgrades (breaking vs non-breaking, bundle impact)
- [ ] Step 4: Implement upgrades one-by-one, run tests after each
- [ ] Step 5: Verify quality gates and bundle size
- [ ] Step 6: Open PR with upgrade rationale
```

## Step 1: Gather Findings

Run dependency audit commands:

```bash
npm audit
npm outdated
```

Categorize by severity:

- Critical
- High
- Moderate
- Low

Cross-reference with project dependency management rules:

- Fix high/critical before merge
- Patch within 48h for critical CVEs

Document findings in a structured table:

| Package | Current Version | Available Version | Severity | CVE IDs |
|---------|----------------|-------------------|----------|----------|
| ... | ... | ... | ... | ... |

## Step 2: Research CVEs

For critical and high vulnerabilities:

- Use **web search** to look up each CVE
- Check exploitability, affected versions, fix version, workarounds
- Review npm advisories and GitHub security advisories for official guidance
- Prioritize: critical first, then high. Moderate/low can be batched.
- Note any packages with no fix available - document mitigation or deferral rationale

## Step 3: Plan Upgrades

Before changing anything:

### Breaking vs Non-Breaking

- Check each package's changelog (npm, GitHub releases)
- Identify major version bumps (breaking changes)
- Review migration guides for breaking changes

### Bundle Impact

- Check bundle size budget from project rules
- Run `npm run build` and measure before/after for each upgrade

### Upgrade Order

1. Security fixes first
2. Non-breaking minor/patch updates
3. Breaking changes (one at a time)

### Risks

- List packages that may require code changes (e.g., major version bumps)
- Note dependencies with breaking API changes

## Step 4: Implement Upgrades

Upgrade **one package at a time** (or one logical group):

1. Run `npm install <package>@<version>`
2. Run `npm install` to update lockfile
3. Run quality gates: `npm run lint && npm run typecheck && npm run test`
4. If tests fail: fix or revert, document required code changes
5. Remove unused dependencies during the pass
6. Commit `package-lock.json` - never use `npm install --no-save`

### Upgrade Protocol

- One package per commit (or one logical group)
- Test after each upgrade
- Revert immediately if breaking
- Document code changes required

## Step 5: Verify

```bash
npm run lint && npm run typecheck && npm run test
npm run build
```

Verify:

- [ ] Bundle size within budget (if defined)
- [ ] No critical or high vulnerabilities remaining: `npm audit`
- [ ] `package-lock.json` is committed

## Step 6: Open PR

Use the project's PR template. Include:

### Upgrade Rationale

Why each package was upgraded (CVE, freshness, feature).

### Breaking Changes

Any code changes required and why.

### Bundle Impact

Before/after gzipped size.

### Test Evidence

All tests pass, no regressions.

### Rollback Plan

If risky (e.g., major version bump).

## Definition of Done

- [ ] No critical or high CVEs remaining
